# Plan 模式 — 重大任务前先对齐方案

与 Cursor **Plan** 模式目标一致：先帮用户想清楚、给可选方案，**用户拍板后再写代码**。  
协作对象默认**技术细节不如 Agent 熟**：少术语、多比喻、选项对比要一眼能懂。

流程硬约束见根 [AGENTS.md](../../AGENTS.md)；本文是操作细则与文档模板。

---

## 何时进入 Plan（必须）

满足**任一**即应先 Plan，**禁止**直接改 `{{SRC_DIR}}/`：

- 新功能 / 新 API / 跨多模块改动（≥3 个目录或影响对外契约）
- 架构或数据模型变更、上线策略、性能/配额/安全类改造
- 需求含糊、有多种实现路径、或用户说「先讨论 / 先方案 / plan 一下」
- 预估改动 > 1 个工作日，或 Agent 无法在一回合内安全交付

**不必 Plan**：明确单点修 bug、改文案、用户已逐步指示且范围清晰、纯 harness/文档维护。

---

## 主 Agent 流程

1. **用用户语言复述目标**（2～4 句），列出 1～3 个待确认问题。
2. **必要时派遣 Subagent**（`explore`）摸清现状。
3. **撰写方案** → `harness/plans/YYYY-MM-DD-主题.md`（模板见 [../plans/_template.md](../plans/_template.md)）。
4. **对话摘要**：目标、方案 A/B、**推荐 + 原因**。
5. **硬停**：等待明确确认。
6. **确认后**：`todo.md` 新增 `task：` → 常规回合。
