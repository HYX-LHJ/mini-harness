# 安全与可靠性检查清单

## 输入/输出安全

- **XSS**：不安全的 HTML 注入、`dangerouslySetInnerHTML`、未转义模板、innerHTML 赋值
- **注入**：通过字符串拼接或模板字面量导致的 SQL/NoSQL/命令/GraphQL 注入
- **SSRF**：用户可控 URL 在未做白名单校验的情况下访问内部服务
- **路径遍历**：用户输入用于文件路径而未做清理（`../` 攻击）
- **原型污染**：JavaScript 中不安全的对象合并（`Object.assign`、与用户输入的展开运算）

## AuthN/AuthZ

- 读/写操作缺失租户或所有权检查
- 新端点缺少认证守卫或 RBAC 强制
- 信任客户端提供的角色/标志/ID
- 访问控制失效（IDOR — 不安全的直接对象引用）
- 会话固定或薄弱的会话管理

## JWT 与令牌安全

- 算法混淆攻击（期望 `RS256` 却接受 `none` 或 `HS256`）
- 弱密钥或硬编码密钥
- 缺失过期时间（`exp`）或未校验
- JWT 载荷中的敏感数据（令牌为 base64，非加密）
- 未校验 `iss`（签发者）或 `aud`（受众）

## 密钥与 PII

- API 密钥、令牌或凭据出现在代码/配置/日志中
- 密钥出现在 git 历史或暴露给客户端的环境变量中
- 过度记录 PII 或敏感载荷
- 错误消息中缺失数据脱敏

## 供应链与依赖

- 未锁定版本导致恶意更新风险
- 依赖混淆（私有包名冲突）
- 从未信任来源或 CDN 导入且未做完整性校验
- 存在已知 CVE 的过时依赖

## CORS 与响应头

- 过于宽松的 CORS（`Access-Control-Allow-Origin: *` 且携带凭据）
- 缺失安全响应头（CSP、X-Frame-Options、X-Content-Type-Options）
- 暴露内部响应头或堆栈跟踪

## 运行时风险

- 无界循环、递归调用或大型内存缓冲区
- 外部调用缺失超时、重试或速率限制
- 请求路径上的阻塞操作（异步上下文中的同步 I/O）
- 资源耗尽（文件句柄、连接、内存）
- ReDoS（正则表达式拒绝服务）

## 密码学

- 弱算法（用于安全目的的 MD5、SHA1）
- 硬编码 IV 或盐值
- 加密未配合认证（ECB 模式、无 HMAC）
- 密钥长度不足

## 竞态条件

竞态条件是导致间歇性故障和安全漏洞的隐蔽缺陷。请特别关注：

### 共享状态访问
- 多个线程/goroutine/异步任务在无同步的情况下访问共享变量
- 并发修改的全局状态或单例
- 懒初始化缺少适当锁（双重检查锁定问题）
- 并发上下文中使用非线程安全集合

### 先检查后执行（TOCTOU）
- `if (exists) then use` 模式而无原子操作
- `if (authorized) then perform`，而授权状态可能已变化
- 先检查文件存在再执行文件操作
- 先检查余额再扣款（金融操作）
- 先检查库存再下单

### 数据库并发
- 缺失乐观锁（`version` 列、`updated_at` 检查）
- 缺失悲观锁（`SELECT FOR UPDATE`）
- 无事务隔离的读-改-写
- 计数器递增非原子（`UPDATE SET count = count + 1`）
- 并发插入导致唯一约束冲突

### 分布式系统
- 共享资源缺失分布式锁
- 领导者选举竞态
- 缓存失效竞态（写入后读到陈旧数据）
- 事件顺序依赖而无适当排序
- 集群操作中的脑裂场景

### 需标记的常见模式
```
# 危险模式：
if not exists(key):       # TOCTOU
    create(key)

value = get(key)          # 读-改-写
value += 1
set(key, value)

if user.balance >= amount:  # 先检查后执行
    user.balance -= amount
```

### 自问
- 「若两个请求同时命中这段代码会怎样？」
- 「该操作是原子的，还是可能被中断？」
- 「这段代码访问哪些共享状态？」
- 「高并发下行为如何？」

## 数据完整性

- 缺失事务、部分写入或不一致的状态更新
- 持久化前校验薄弱（类型强制转换问题）
- 可重试操作缺失幂等性
- 并发修改导致更新丢失
