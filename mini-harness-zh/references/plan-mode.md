# Plan 模式 — 重大任务前先对齐方案

与 Cursor **Plan** 模式目标一致：先帮用户想清楚、给可选方案，**用户拍板后再写代码**。  
协作对象默认**技术细节不如 Agent 熟**：少术语、多比喻、选项对比要一眼能懂。

流程硬约束见项目 `AGENTS.md`；本文是操作细则与文档模板。

---

## 何时进入 Plan（必须）

满足**任一**即应先 Plan，**禁止**直接改业务代码目录：

- 新功能 / 新 API / 跨多模块改动（≥3 个目录或影响对外契约）
- 架构或数据模型变更、上线策略、性能/配额/安全类改造
- 需求含糊、有多种实现路径、或用户说「先讨论 / 先方案 / 头脑风暴 / plan 一下」
- 预估改动 > 1 个工作日，或 Agent 无法在一回合内安全交付

**不必 Plan**：明确单点修 bug、改文案、用户已逐步指示且范围清晰、纯 harness/文档维护。

**与 Cursor Plan 的关系**：可建议用户切换到 Cursor **Plan** 模式；若仍在 **Agent** 模式，须遵守「未确认不写业务代码」。

---

## 主 Agent 流程

1. **用用户语言复述目标**（2～4 句），列出 1～3 个待确认问题（选择题优先）。
2. **必要时派遣 Subagent**（`explore`）摸清现状；结论翻译成用户能懂的要点。
3. **撰写方案** → `harness/plans/YYYY-MM-DD-主题.md`（模板见 `harness/plans/_template.md`）。
4. **对话摘要**：目标、方案 A/B、**推荐 + 原因**、请用户选哪项。
5. **硬停**：等待明确确认。未确认前仅可改 `harness/plans/`、`todo.md`（Plan 子项）、`DECISIONS.md`（架构取舍）。
6. **确认后**：`todo.md` 新增实现 `task：` → 进入常规回合（门禁、tdd、实现）。

---

## 对用户说话的原则

- **先结论后细节**；用「我建议」「您只需要决定」降低压力。
- 术语第一次出现用括号白话解释。
- 不给假选项；若只有一条路，如实说明。
- 小任务直接做；大任务才 Plan。

---

## 与其它产物的关系

| 产物 | 区别 |
|------|------|
| **plans/** | 实施**前**、给用户选的方案 |
| **DECISIONS.md** | 已确认后的架构约束 |
| **code_review/** | 实施**后**的代码审查 |
| **todo.md** | Plan 确认后的可执行 checklist |

用户确认 Plan 后，若涉及长期约束，应在 `DECISIONS.md` 追加摘要。
