# SOLID 异味提示

## SRP（单一职责）

- 文件承担无关关注点（例如，HTTP + 数据库 + 领域规则在同一文件中）
- 内聚性低或有多重变更理由的大型类/模块
- 编排许多无关步骤的函数
- 对系统了解过多的上帝对象
- **自问**：「该模块发生变更的单一理由是什么？」

## OCP（开闭原则）

- 添加新行为需要编辑许多 switch/if 分支
- 功能增长需要修改核心逻辑而非扩展
- 缺少用于变体的插件/策略/钩子扩展点
- **自问**：「能否在不改动现有代码的情况下添加新变体？」

## LSP（里氏替换）

- 子类检查具体类型，或对基类方法抛出异常
- 重写方法削弱前置条件或加强后置条件
- 子类忽略或对父类行为空操作（no-op）
- **自问**：「能否在调用方不知情的情况下替换任意子类？」

## ISP（接口隔离）

- 接口方法很多，但实现者大多未使用
- 调用方为狭窄需求依赖宽泛接口
- 接口方法的空实现/桩实现
- **自问**：「所有实现者是否都使用所有方法？」

## DIP（依赖倒置）

- 高层逻辑依赖具体的 IO、存储或网络类型
- 硬编码实现而非抽象或依赖注入
- 将业务逻辑与基础设施耦合的导入链
- **自问**：「能否在不修改业务逻辑的情况下替换实现？」

---

## 常见代码异味（超越 SOLID）

| 异味 | 迹象 |
|-------|-------|
| **长方法** | 函数超过 30 行，多层嵌套 |
| **依恋情结** | 方法使用其他类的数据多于自身 |
| **数据泥团** | 同一组参数反复一起传递 |
| **基本类型偏执** | 使用字符串/数字而非领域类型 |
| **散弹式修改** | 一次变更需要修改多个文件 |
| **发散式变化** | 一个文件因多种无关理由而变更 |
| **死代码** | 不可达或从未被调用的代码 |
| **过度设计** | 为假想的未来需求引入抽象 |
| **魔法数字/字符串** | 硬编码值而无命名常量 |

## 重构启发式

1. **按职责拆分，而非按大小** — 小文件仍可能违反 SRP
2. **仅在需要时引入抽象** — 等到第二个用例再抽象
3. **保持增量重构** — 移动前先隔离行为
4. **先保持行为不变** — 重构前先补充测试
5. **按意图命名** — 若命名困难，抽象可能有问题
6. **优先组合而非继承** — 继承会造成紧耦合
7. **使非法状态无法表示** — 用类型强制不变量
