# 测试驱动开发

## 理念

**核心原则**：测试应通过公共接口验证行为，而非实现细节。代码可以完全重写；测试不应随之失效。

**好的测试**是集成风格的：它们通过公共 API 走真实代码路径。它们描述系统**做什么**，而非**怎么做**。好的测试读起来像规格说明——「用户可以使用有效购物车结账」准确说明了系统具备的能力。这类测试能经受重构，因为它们不关心内部结构。

**坏的测试**与实现耦合。它们 mock 内部协作者、测试私有方法，或通过外部手段验证（例如直接查询数据库，而不是使用接口）。警示信号：重构后测试失败，但行为并未改变。若重命名内部函数后测试失败，说明那些测试验证的是实现，而非行为。

示例见 [references/tests.md](references/tests.md)，
mock 指南见 [references/mocking.md](references/mocking.md)。

## 反模式：水平切片

**不要先写完全部测试，再写全部实现。** 这叫「水平切片」——把 RED 当成「写所有测试」，把 GREEN 当成「写所有代码」。

这会产生**糟糕的测试**：

- 批量写出的测试验证的是**想象**的行为，而非**实际**行为
- 你最终测试的是事物的**形态**（数据结构、函数签名），而非面向用户的行为
- 测试对真实变化不敏感——行为坏了仍通过，行为正常却失败
- 你跑在认知前面，在理解实现之前就锁定了测试结构

**正确做法**：用 tracer bullet 做垂直切片。一个测试 → 一个实现 → 重复。每个测试都回应上一轮回合的收获。因为你刚写完代码，你清楚哪些行为重要、如何验证。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域词汇表，使测试名称与接口用语与项目语言一致，并尊重你所改动区域的 ADR。

在写任何代码之前：

- [ ] 阅读 `harness/todo.md` 中的 **验收标准（AC）**（主来源；若有 plan 链接则一并阅读）
- [ ] **确认 todo 中 `- [x] **AC 已确认**` 已勾选**；未勾选则 **停止**，先与用户核对 AC 文案与场景，不得编写测试
- [ ] 与用户确认需要哪些接口变更
- [ ] 与用户确认要测试哪些行为（排优先级）
- [ ] 识别 [深度模块](references/deep-modules.md) 的机会（小接口、深实现）
- [ ] 为[可测试性](references/interface-design.md)设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的认可

问：「公共接口应该长什么样？哪些行为最值得测试？」

**你无法测试一切。** 与用户确认哪些行为最重要。把测试精力集中在关键路径和复杂逻辑上，而非每个可能的边界情况。

### 2. Tracer Bullet

写一个测试，验证系统的一件事：

```
RED:   为第一个行为写测试 → 测试失败
GREEN: 写最少代码使其通过 → 测试通过
```

这就是你的 tracer bullet——证明端到端路径可行。

### 3. 增量循环

对每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最少代码使其通过 → 通过
```

规则：

- 一次只写一个测试
- 只写足以通过当前测试的代码
- 不要预判后续测试
- 让测试聚焦可观察的行为

### 4. 重构

全部测试通过后，寻找[重构候选](references/refactoring.md)：

- [ ] 提取重复
- [ ] 加深模块（把复杂度藏在简单接口之后）
- [ ] 在合适处应用 SOLID 原则
- [ ] 思考新代码对既有代码的启示
- [ ] 每步重构后运行测试

**不要在 RED 时重构。** 先回到 GREEN。

## 每轮检查清单

```
[ ] 测试描述行为，而非实现
[ ] 测试只使用公共接口
[ ] 测试能经受内部重构
[ ] 代码对本测试而言是最少的
[ ] 未添加投机性功能
```
