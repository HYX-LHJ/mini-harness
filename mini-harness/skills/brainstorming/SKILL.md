# Agent Harness 头脑风暴

将不清晰或影响重大的请求转化为已达成一致、可实施的方向，同时避免把每个小任务都变成设计项目。

## 首先判断是否需要头脑风暴

当以下至少一项成立时，使用完整工作流：

- 用户明确要求头脑风暴、讨论、设计或规划。
- 目标、成功标准或范围不清晰。
- 存在两种或以上真正不同的方案，且各有有意义的取舍。
- 变更影响外部契约、数据模型、安全、并发、发布行为或多个模块边界。

对于明显的拼写错误、文档修正、范围狭窄的 bug 修复或用户已明确指定的变更，简要说明假设后回到正常 harness 流程。不要人为制造备选方案。

## 工作流

### 1. 恢复上下文

阅读 `harness/PROGRESS.md`、`harness/todo.md`、相关代码，以及 `harness/DECISIONS.md` 中的有效约束。只阅读决策所需的内容。

### 2. 框定决策

总结：

- 用户想要达成什么；
- 已知信息有哪些；
- 尚不确定的是什么；
- 什么结果算成功。

若请求涉及多个独立系统，在细化实现细节前先拆分。

### 3. 提出聚焦的问题

一次只问一个阻塞性问题。当确实存在可选方案时，优先给出少量具体选项。不要问能从仓库中安全查明的答案。

聚焦目的、约束、兼容性、可接受风险与成功标准。

### 4. 比较方案

当确实存在备选方案时，呈现两到三种方案，包含：

- 核心思路；
- 收益；
- 成本或风险；
- 各方案适用场景；
- 结合用户优先级的明确推荐。

若只有一条可行路径，直接说明，并解释剩余实现选择，而非编造虚假选项。

### 5. 呈现拟议设计

按任务规模调整设计详略。只覆盖相关部分：

- 范围与非目标；
- 受影响的边界或组件；
- 数据流或控制流；
- 失败处理；
- 兼容性与迁移；
- 测试与验收证据。

当决策具有实质影响时，在修改运行时代码前先征求用户确认。

### 6. 仅在有用时记录

当设计体量较大、影响后续工作，或需要持久化审批记录时，写入 `harness/plans/YYYY-MM-DD-主题.md`。小决策可留在对话与 todo 中。

使用 `harness/docs/plan-mode.md` 中的结构。不要自动提交。

### 7. 过渡

获得批准后：

1. 在 `harness/todo.md` 中登记实现工作，并写入 **验收标准（AC）** 表（从 plan 同步，或对小任务直接在此撰写）；
2. 加载相关的内置开发与测试 Skills；
3. 继续按正常 harness 工作流推进。

## 原则

- 仪式感应与不确定性和风险成正比。
- 优先依据仓库证据，而非臆测性提问。
- 使用用户的语言，并通俗解释不熟悉的取舍。
- 给出方向性推荐，而非仅罗列选项。
- 遵循 YAGNI，避免无关重构。
- 审批关口保护实质性决策，而非琐碎编辑。
